# Unified Monorepo Structure

Kest uses a flat, high-visibility monorepo for its core codebase.

The reason is simple: the tool, the platform logic, and the interface should evolve around the same product meaning instead of drifting apart across separate repositories.

## Current Structure

```text
cli/   - Kest CLI
api/   - platform backend
web/   - web console
```

## Why This Structure Exists

This layout solves three practical problems.

### 1. It keeps tool and platform semantics aligned

If the CLI and backend are too disconnected, request models, execution results, error semantics, and AI context tend to drift.

One repository reduces that risk.

### 2. It supports cross-layer product work

Many Kest features naturally span multiple layers. For example:

- the CLI captures and executes test flows
- the API persists results and manages collaboration logic
- the web app visualizes flows, history, and team activity

Keeping those layers together makes coordinated changes easier.

### 3. It works better with AI-assisted development

When AI is part of the development loop, full-repository context matters. A unified monorepo makes it easier for AI tools to understand:

- how CLI behavior maps to backend data structures
- how backend capabilities surface in the UI
- which layers a product change needs to touch

## What Each Directory Owns

### `cli/`

The CLI is the execution surface. It is optimized for:

- speed
- markdown-native flows
- terminal automation

### `api/`

The backend owns:

- team collaboration
- data persistence
- AI diagnosis and platform logic

### `web/`

The web console owns:

- flow visualization
- team activity views
- interactive entry points for platform features

This structure is not cosmetic. It exists to keep the product coherent across all three layers.

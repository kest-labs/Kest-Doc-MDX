# 统一 Monorepo 结构

Kest 使用扁平、可见性高的 Monorepo 结构组织核心代码。

目标很直接：让工具、平台逻辑和界面层围绕同一个产品语义保持同步，而不是分散成多个逐渐漂移的仓库。

## 当前结构

```text
cli/   - Kest CLI
api/   - 平台后端
web/   - Web Console
```

## 为什么要这样组织

这套结构主要解决三类问题。

### 1. 工具和平台语义保持一致

如果 CLI 和平台后端分离得太远，变量模型、执行结果、错误语义和 AI 分析上下文就很容易出现理解偏差。

统一仓库可以降低这种漂移。

### 2. 更适合跨层联动开发

Kest 的很多能力天然跨越多个层次，例如：

- CLI 采集测试结果
- API 持久化并做协作编排
- Web 展示运行历史和团队状态

这些需求放在同一个代码库中更容易整体推进。

### 3. 更适合 AI 协作

当 AI 参与开发时，完整的仓库上下文很重要。统一 Monorepo 能让 AI 更容易理解：

- 命令行行为如何映射到后端数据结构
- 后端能力如何在前端中可视化呈现
- 某个产品能力需要改动哪些层

## 每个目录承担什么职责

### `cli/`

CLI 是 Kest 的执行入口，强调：

- 高性能
- Markdown 原生
- 适合终端自动化

### `api/`

后端负责：

- 团队协作能力
- 数据持久化
- AI 诊断和平台逻辑

### `web/`

Web Console 负责：

- 测试流可视化
- 团队活动展示
- 平台能力的交互入口

这套结构不是为了“看起来整齐”，而是为了保证产品能力在三个层面上持续对齐。

# AI 原生工作流

传统 API 工具大多是围绕“手动点点点”设计的。

Kest 的设计前提不同：AI 已经成为日常开发流程的一部分，所以测试系统本身也应该对 AI 友好。

## Markdown 原生

Kest 使用 `.flow.md` 作为流程描述格式。

这意味着一份测试流可以同时满足三类需求：

- 人类可以像读文档一样快速理解流程
- AI 可以直接读取、生成和改写内容
- 代码仓库可以像管理普通文本一样管理测试资产

相比表单式工具，这种表达方式更稳定，也更适合审查和版本管理。

## 零拷贝链式传递

接口测试很少只有一步。

你通常会经历这样的过程：

1. 登录
2. 拿到 token
3. 带着 token 请求受保护接口
4. 对结果做断言

Kest 的链式变量捕获就是为这个过程服务的。一个请求拿到的数据可以直接成为下一个请求的输入，不需要人工复制粘贴。

## AI 诊断

当某个测试失败时，`kest why` 的目标不是只告诉你“失败了”，而是结合本地请求历史帮助你定位更接近根因的解释。

这对 AI 协作很重要，因为真正有价值的不是报错本身，而是：

- 失败发生在链路的哪一段
- 是请求构造问题、数据问题还是环境问题
- 哪个上游步骤最可能导致当前异常

## 快照测试

很多接口回归并不来自状态码错误，而是来自响应结构、关键字段或返回内容的变化。

Kest 提供快照测试能力，是为了让你在不写大量样板断言的前提下，仍然能够快速发现回归。

## 为什么这套方式适合 Vibe Coding

所谓 Vibe Coding，并不只是“让 AI 写代码”，而是把自然语言、代码、命令和验证过程放在一条更连续的链路里。

Kest 的工作流正好补上了 API 测试这一环：

- AI 可以生成测试流
- 开发者可以在终端中立即执行
- 结果可以继续反馈给 AI 调整实现
- 团队可以在平台上沉淀最终流程
